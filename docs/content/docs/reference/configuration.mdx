## Config file location

```
~/.opentabs/config.json
```

Override the config directory with the `OPENTABS_CONFIG_DIR` environment variable. The file is created automatically by `opentabs start` on first run with `0600` permissions.

## Example config

```json
{
  "localPlugins": [
    "../plugins/my-plugin",
    "/Users/you/projects/opentabs-plugin-jira"
  ],
  "tools": {
    "slack_delete_message": false
  },
  "browserToolPolicy": {
    "browser_execute_script": false
  },
  "permissions": {
    "trustedDomains": ["localhost", "127.0.0.1"],
    "sensitiveDomains": ["*.bank.example.com"],
    "toolPolicy": {
      "browser_screenshot_tab": "allow"
    },
    "domainToolPolicy": {
      "app.example.com": {
        "browser_click_element": "allow",
        "browser_execute_script": "deny"
      }
    }
  }
}
```

## Field details

### localPlugins

An array of filesystem paths to local plugin directories. These are plugins under active development on your machine. Paths starting with `./` or `../` are resolved relative to the config directory (`~/.opentabs/`). Absolute paths and `~/` paths are also supported.

Local plugins are registered automatically when you first run `opentabs-plugin build` in a plugin directory. You can also add them manually.

npm plugins (installed globally via `npm install -g`) are auto-discovered from global `node_modules` — they do not need to be listed here. Local plugins override npm plugins of the same name.

### tools

Disable specific tools without removing the plugin:

```json
{
  "tools": {
    "slack_delete_message": false,
    "slack_send_message": false
  }
}
```

Only include entries for tools you want to disable — absent keys default to enabled. Use the CLI to toggle tools:

```bash
opentabs config set tool.slack_send_message disabled
opentabs config set tool.slack_send_message enabled
```

### secret

The authentication secret is stored in `~/.opentabs/extension/auth.json`, not config.json:

```json
{
  "secret": "<64-character hex string>"
}
```

Auto-generated on first run. Used for authentication across all transports:

- **`/mcp`** and admin endpoints: `Authorization: Bearer <secret>` header
- **`/ws`**: `Sec-WebSocket-Protocol` header (keeps the secret out of URLs and logs)

Retrieve it for MCP client configuration:

```bash
opentabs config show --json --show-secret | jq -r .secret
```

The `--show-secret` flag is required to output the full secret value. Without it, the secret is truncated. The source of truth is always auth.json.

### browserToolPolicy

Disable specific browser tools without affecting plugin tools:

```json
{
  "browserToolPolicy": {
    "browser_execute_script": false,
    "browser_get_cookies": false
  }
}
```

Only include entries for tools you want to disable — absent keys default to enabled. Use the CLI to toggle browser tools:

```bash
opentabs config set browser-tool.browser_execute_script disabled
opentabs config set browser-tool.browser_execute_script enabled
```

### permissions

Controls the browser tool confirmation system — which tools require human approval and on which domains. The `permissions` object has four sub-fields:

      <Table.Cell>Per-tool policy overrides. Keys are browser tool names. `allow` skips confirmation, `ask` requires it, `deny` blocks the tool entirely.</Table.Cell>
    </Table.Row>
    <Table.Row>

      <Table.Cell className="*:text-xs">`{}`</Table.Cell>
      <Table.Cell>Per-domain per-tool overrides. Most specific — takes priority over all other policies. Keys are domains, values are tool-to-permission maps.</Table.Cell>
    </Table.Row>
  </Table.Body>
</Table>

The permission evaluation uses a first-match-wins order:

1. `domainToolPolicy[domain][tool]` — most specific, checked first
2. `sensitiveDomains` match → forces `ask`
3. `toolPolicy[tool]` — global per-tool override
4. Tool tier default (`observe` → `allow`, `interact`/`sensitive` → `ask`)
5. `trustedDomains` match → upgrades `ask` to `allow` (does **not** override `deny`)

Example — allow screenshots everywhere, require confirmation for script execution on a specific domain, and mark banking sites as sensitive:

```json
{
  "permissions": {
    "trustedDomains": ["localhost", "127.0.0.1", "*.internal.company.com"],
    "sensitiveDomains": ["*.bank.example.com"],
    "toolPolicy": {
      "browser_screenshot_tab": "allow"
    },
    "domainToolPolicy": {
      "app.example.com": {
        "browser_execute_script": "deny"
      }
    }
  }
}
```

### skipConfirmation

Set to `true` to bypass all browser tool confirmation prompts. This disables the human-in-the-loop safety check entirely — all tools run without asking for approval, even on sensitive domains.

**This is dangerous and intended only for CI/testing environments.** In normal usage, leave this unset and use the `permissions` object to fine-tune which tools require confirmation.

Can also be enabled via the `OPENTABS_SKIP_CONFIRMATION` environment variable or the `--dangerously-skip-confirmation` flag on the MCP server binary (not the `opentabs` CLI). Prefer the env var for portability.

## Plugin discovery

The MCP server discovers plugins from two sources:

1. **npm auto-discovery** — scans global `node_modules` for packages matching `opentabs-plugin-*` and `@*/opentabs-plugin-*`. Install plugins with `npm install -g <package>`.
2. **Local plugins** — directories listed in the `localPlugins` array. Registered automatically by `opentabs-plugin build` on first build.

Local plugins override npm plugins of the same name. Discovery runs at server startup and whenever `POST /reload` is called.

## Dev vs production mode

The MCP server runs in **production mode** by default. Dev mode is intended for platform contributors working on the server itself.

### Production mode (default)

```bash
opentabs start
```

- Discovers plugins once at startup
- No file watchers — minimal background filesystem activity
- `POST /reload` is available for on-demand rediscovery (used by `opentabs-plugin build`)

### Dev mode (contributors only)

```bash
npm run dev
```

- **File watchers** monitor local plugin `dist/` directories — auto-reloads when plugins are rebuilt
- **Config watching** detects changes to `config.json` and re-discovers plugins
- **Hot reload** — rebuilt server files take effect without restarting

Use `npm run dev:mcp` for a lightweight alternative that runs the MCP server only (no tsc watch or extension build).

Dev mode is intended for platform contributors working on the server source code. Normal users and plugin developers use production mode.

The `/health` endpoint includes a `mode` field (`"dev"` or `"production"`) to verify which mode the server is running in.

## Config directory structure

<ConfigDirectory />

Both `config.json` and `auth.json` are created with `0600` permissions (owner read/write only) to protect the shared secret.

The `extension/` directory is managed by `opentabs start`. Load it as an unpacked extension from `chrome://extensions`. The `adapters/` subdirectory holds the built adapter IIFE files for each discovered plugin — these are what the extension injects into matching tabs.

Config writes use an atomic pattern: content is written to a temp file then renamed over the target. A mutex serializes concurrent writes.

## Legacy config migration

If your config still has the old `plugins` array or `npmPlugins` field, the server migrates automatically:

- **`plugins` array** — local paths (starting with `./`, `../`, `/`, `~/`) are moved to `localPlugins`. npm package names are dropped (npm plugins are now auto-discovered).
- **`npmPlugins` array** — entries are dropped with a log notice (npm plugins are auto-discovered from global `node_modules`).

Update your config to use `localPlugins` to suppress migration log messages.

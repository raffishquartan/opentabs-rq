OpenTabs has three main components: the **MCP server** (discovers plugins and dispatches requests), the **Chrome extension** (injects plugin adapters into browser tabs), and **plugins** (define tools, resources, and prompts that run in the page context). All communication uses JSON-RPC 2.0.

## MCP Server

The MCP server runs on `localhost:9515` (configurable via the `PORT` env var). The `opentabs` CLI also accepts `OPENTABS_PORT` as a user-facing convenience — it reads the value and passes it to the server process as `PORT` when spawning. It is an HTTP + WebSocket server (`node:http` + `ws`) with two responsibilities:

1. **Plugin discovery** — resolves plugins from npm and local paths, loads their metadata and adapter bundles, and builds an immutable registry
2. **Request dispatch** — registers plugin tools, resources, and prompts as MCP capabilities, dispatches calls to the Chrome extension via WebSocket, and relays results back to MCP clients

### Discovery Pipeline

Plugin discovery is a five-phase pipeline in `platform/mcp-server/src/discovery.ts`:

1. **Discover npm plugins** — scans global `node_modules` for packages matching `opentabs-plugin-*` and `@*/opentabs-plugin-*`
2. **Resolve local plugins** — reads filesystem paths from `config.localPlugins` and resolves them to absolute directories
3. **Load all plugins** — reads `package.json` (with `opentabs` metadata field), `dist/adapter.iife.js`, and `dist/tools.json` for each plugin in parallel; assigns a trust tier to each: Official (`@opentabs-dev/*`), Community (any other npm package), or Local (filesystem path)
4. **Merge** — local plugins override npm plugins of the same name
5. **Build registry** — constructs an immutable `PluginRegistry` with O(1) lookup maps for tools, resources, and prompts

Errors in any phase are collected but non-fatal — other plugins still load. Failed plugins are tracked in the registry for diagnostic purposes (visible in `/health`).

### Plugin Registry

The `PluginRegistry` (`platform/mcp-server/src/registry.ts`) is immutable (`Object.freeze()`). It maintains four internal `Map`s:

- `plugins` — name → `RegisteredPlugin`
- `toolLookup` — prefixed tool name → `ToolLookupEntry` (with pre-compiled Ajv validators)
- `resourceLookup` — prefixed URI → `ResourceLookupEntry`
- `promptLookup` — prefixed prompt name → `PromptLookupEntry`

When plugins change, a new registry is built and atomically swapped on the server state. Ajv validators are compiled during registry construction, so tool dispatch has O(1) lookup with no schema compilation at call time.

### Hot Reload Architecture

In dev mode, the MCP server runs behind a thin dev proxy (`src/dev-proxy.ts`). The proxy holds all HTTP and WebSocket connections while watching `dist/` for `.js` file changes. On change (debounced 300ms), the proxy kills the current worker process and forks a new one. The new worker starts on an ephemeral port, signals its actual port to the proxy via IPC, and the proxy resumes forwarding traffic.

On each hot reload cycle:

1. The dev proxy detects changed `.js` files in `dist/`
2. The current worker process is killed and a new one is forked
3. The new worker re-reads config, re-discovers plugins, and builds a new registry
4. MCP handlers are registered on new sessions
5. A `sync.full` is sent to the extension with the updated plugin list
6. MCP clients are notified via `tools/list_changed`

## Chrome Extension

The Chrome extension is a Manifest V3 extension with three main parts: a **background service worker**, an **offscreen document**, and a **side panel**.

### Persistent WebSocket

MV3 service workers can be suspended at any time. To maintain a persistent WebSocket, the extension uses an offscreen document — a hidden HTML page with a stable execution context. The offscreen document:

- Connects to `ws://localhost:9515/ws`
- Authenticates via the `Sec-WebSocket-Protocol` header (keeps the secret out of URLs and logs)
- Runs ping/pong keepalive every 15 seconds with a 5-second timeout
- Reconnects with exponential backoff (1s → 2s → 4s → ... → 30s cap)

A Chrome alarm re-ensures the offscreen document exists under memory pressure.

### Adapter Injection

Plugin adapters are injected into matching tabs as IIFE bundles using `chrome.scripting.executeScript()` with `files` and `world: 'MAIN'`:

1. A tab loads and reaches `status === 'complete'`
2. The background script queries matching tabs by URL pattern
3. For each matching tab without an adapter, it injects the plugin's IIFE from `~/.opentabs/extension/adapters/<plugin>.js`
4. The adapter registers itself at `globalThis.__openTabs.adapters[pluginName]`

File-based injection bypasses page Content Security Policy (CSP). Because adapter files are bundled with the extension, Chrome treats them as extension resources exempt from page CSP.

### Dispatch Flow

When an AI agent calls a tool:

1. The MCP server validates the input against the pre-compiled JSON Schema
2. The server sends a `tool.dispatch` JSON-RPC request over WebSocket
3. The extension finds matching tabs using the plugin's URL patterns
4. Tabs are ranked: active tab in focused window > active tab in any window > any tab in focused window > other tabs
5. The extension calls `adapter.handle(toolName, input)` in the MAIN world via `chrome.scripting.executeScript()`
6. The result flows back through WebSocket to the MCP server, which returns it to the AI agent

Resource reads (`resource.read`) and prompt gets (`prompt.get`) follow the same pipeline. They do not support progress reporting or lifecycle hooks.

## Security

### Authentication

- **`/mcp`**: Bearer token in the `Authorization` header — the 64-character hex `secret` from `~/.opentabs/extension/auth.json`
- **`/ws`**: Auth via `Sec-WebSocket-Protocol` — the client sends `['opentabs', '<secret>']`, server echoes `'opentabs'` as the accepted subprotocol
- **`/reload` and `/extension/reload`**: Bearer token required, plus rate limiting (10 requests/minute)
- **`/ws-info` and `/audit`**: Bearer token required, no rate limiting

### CORS Protection

The server rejects any HTTP request with an `Origin` header unless it matches `chrome-extension://`. Legitimate MCP clients do not send Origin headers, so this prevents DNS rebinding attacks without affecting normal usage.

### Adapter Integrity

The `opentabs-plugin build` CLI computes a SHA-256 hash of the adapter IIFE and embeds it as `__adapterHash`. The extension verifies this hash after injection to detect partial file writes or tampering.

### Output Sanitization

Tool outputs are sanitized to remove keys that could cause prototype pollution. Error messages have internal file paths, IP addresses, and URLs stripped before being returned to clients.

Tools not in the tier map default to `interact` as a safe middle ground.

**Permission evaluation order** — first match wins:

1. **`domainToolPolicy[domain][tool]`** — per-domain per-tool override (most specific)
2. **`sensitiveDomains` match** → forces `ask`
3. **`toolPolicy[tool]`** — global per-tool override
4. **Tool tier default** — `observe` → `allow`, `interact` → `ask`, `sensitive` → `ask`
5. **`trustedDomains` match** — upgrades `ask` to `allow` (does **not** override `deny`)

Session permission rules (created by "Allow Always" actions) are checked before the evaluation chain. If a matching session rule exists, the tool is auto-allowed without running the chain. The `skipConfirmation` config flag bypasses the entire evaluation — all tools are auto-allowed.

### Confirmation System

When a browser tool evaluates to `ask`, the server requests human approval through the Chrome extension's side panel:

1. Server sends `confirmation.request` to the extension with a UUID, tool name, domain, tab ID, truncated parameter preview, and timeout
2. The side panel displays an approval dialog
3. The user chooses one of three decisions: `allow_once`, `allow_always` (with a scope), or `deny`
4. The extension sends `confirmation.response` back with the decision
5. Server proceeds with the tool call or returns a denial error

For `allow_always`, the user selects a scope that determines how broad the session rule is:
